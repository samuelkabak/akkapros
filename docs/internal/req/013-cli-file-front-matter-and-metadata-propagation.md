# Requirement: CLI File Front Matter and Metadata Propagation

REQ-ID: REQ-013
Status: Draft
Priority: High
Impact: Mutative
Created: 2026-03-27
Updated: 2026-03-27
---

# Summary

The system shall add structured front matter to files generated by CLI pipeline
steps so metadata, validation facts, and stage-specific data can travel with
the file content itself. This replaces the discarded lexical-output approach
for propagating construct-state and related metadata.

The contract shall use YAML front matter for text outputs and an equivalent
top-level `frontmatter` object for metrics JSON output. Metrics CSV is outside
this change and has no embedded front matter contract here.

For compact validation-oriented stage data, the preferred canonical field name
for explicit user word links is `explicit_word_link_count`.

---

# Motivation

Current pipeline files contain content only. Downstream stages therefore lack a
uniform way to validate file format/version, preserve stable file identity, or
consume upstream metadata without introducing special-purpose outputs or custom
pairing rules.

Embedding structured metadata in generated files provides a general solution
for provenance, validation, and controlled metadata propagation across the
whole pipeline.

# Acceptance Criteria
*Verifiable conditions that must be met. Use Given/When/Then format where appropriate.*

- [ ] Given a CLI-generated text output, when it is written, then the file
      begins with a YAML front matter block delimited by `---`, followed by one
      empty line and then the content body.
- [ ] Given a metrics JSON output, when it is written, then it contains a
      top-level `frontmatter` object that is structurally equivalent to the
      YAML front matter contract used by text outputs.
- [ ] Given a metrics CSV output, when it is written, then it contains no
      embedded front matter block and remains a plain CSV artifact.
- [ ] Given any front matter produced by the pipeline, when keys are emitted,
      then all keys use lowercase snake_case.
- [ ] Given a generated file, when front matter is emitted, then it includes
      `package.name`, `package.version`, `pipeline`, `step`, `file.id`,
      `file.title`, `file.format`, `file.version`, and `file.date`.
- [ ] Given a generated file covered by this requirement, when `pipeline` is
      emitted, then its value is `pipeline`.
- [ ] Given a generated file covered by this requirement, when `file.format`
      is emitted, then it equals the filename suffix before the extension,
      including `metrics` for both `_metrics.txt` and `_metrics.json`.
- [ ] Given a generated file derived from another pipeline file, when front
      matter is emitted, then `metadata.input_file_id` contains the immediate
      input file identifier.
- [ ] Given CLI options for a stage, when `metadata.options` is emitted, then
      it excludes filesystem-bearing options such as input filename and output
      directory, converts option names to snake_case, preserves inherited
      upstream option values, and includes default values only when the
      effective value is neither an empty string nor `false`.
- [ ] Given stage-specific data to propagate, when `metadata.data` is emitted,
      then it nests data under stage keys in pipeline order (`atfparse`,
      `syllabify`, `prosody`, `metrics`, `print`) and omits empty or not-yet-
      reached stage blocks.
- [ ] Given stage-specific counters for explicit user word links, when emitted,
      then the canonical field name is `explicit_word_link_count`.
- [ ] Given a file produced from an ATF source, when downstream files are
      written, then `file.title` is propagated from the source title captured
      from the ATF header line.
- [ ] Given an input file with front matter, when any downstream CLI stage
      reads it, then the stage can parse the front matter and validate the
      declared file format and file-version before processing the content body.
- [ ] Given an input line that ends with attached `-` or `+` and the lexical or
      prosodic group continues on the following physical line without
      intervening whitespace, when a pipeline stage reads it, then the stage
      fails with a clear validation error and does not auto-merge the lines.
- [ ] Given the front matter contract, when format templates and versions are
      maintained, then they are defined centrally in `src/akkapros/lib/frontmatter.py`.
- [ ] Given this requirement is implemented, when documentation is updated,
      then user-facing docs describe the new file contract and migration
      expectations and developer-facing docs describe the shared schema,
      validation rules, and maintenance location.
- [ ] Given this requirement is implemented, when tests are updated, then unit
      coverage is added to built-in `run_tests()` where available and pytest
      coverage includes both focused validation tests and end-to-end pipeline
      integration coverage for the front matter contract.

# User Story (optional)
> As a researcher running a multi-step Akkadian prosody pipeline, I want each
> generated file to carry validated metadata and stage data so that downstream
> tools can consume provenance directly without relying on ad hoc auxiliary
> outputs.

---

# Interface Notes
- Input: pipeline files produced by earlier CLI stages, with or without front
  matter during migration.
- Output: text files with YAML front matter, metrics JSON with `frontmatter`,
  and no front matter contract for metrics CSV.
- Affected components: `src/akkapros/cli/atfparser.py`,
  `src/akkapros/cli/syllabify.py`, `src/akkapros/cli/prosmaker.py`,
  `src/akkapros/cli/metricalc.py`, printer/format CLI modules, and the shared
  front matter support library `src/akkapros/lib/frontmatter.py`.

# Non-Goals
- This requirement does not define production-code implementation details for
  YAML parsing, UUID generation, or serializer internals.
- This requirement does not retain the discarded `_lex.txt` propagation path.
- This requirement does not define a front matter contract for metrics CSV.
- This requirement does not authorize silent repair by merging physical lines
      that were split after attached terminal `-` or `+` markers.

# Security / Safety Considerations
- Front matter must not leak filesystem-sensitive values such as source paths
  or output directories.
- Validation logic must treat front matter as untrusted input and fail
  predictably when required fields or supported versions are missing.
