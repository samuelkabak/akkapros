# Governance Guidelines — Quick Reference

**Purpose:** Compressed summary of governance rules for agent use. Do NOT read all CR/REQ/ADR/REVIEW docs before working. Read this file first, then read only the directly relevant records.

**Update rule:** Update this file whenever a new governance rule is introduced.

---

## 1. Document Types

| Type | Purpose | Statuses |
|------|---------|----------|
| ADR | Architecture decisions (motivation, alternatives, decision, consequences) | Proposed → Accepted / Rejected |
| CR | Implementation work (scope, acceptance criteria, verification) | Draft → Approved → Done / Rejected / Blocked |
| REQ | Testable requirements | Draft → Approved → Implemented / Rejected |
| REVIEW | Verification and review outcomes | Draft → Done |

## 2. Reading Budget (CRITICAL — saves tokens)

- **Start with:** `docs/internal/GUIDELINES.md` (this file)
- **Then read:** the target CR + its directly referenced ADRs/REQs/CRs only
- **Do NOT:** scan governance history from `001` upward, read all previous CRs, or bulk-load surrounding records
- **Expand only:** when a named ambiguity or dependency cannot be resolved with current context
- **Index pages:** `docs/internal/*/index.md` for quick lookup of what exists

## 3. Precedence

- Newer higher-numbered records override older accepted behavior
- After `999`, identifiers continue: `A00`..`A99`, `B00`..`B99`, etc.
- Preserve older records as history; do not rewrite them
- A CR is NOT blocked merely because it conflicts with an older accepted record — follow the newer contract

## 4. CR Workflow

- **Executability gate:** Before implementing, check if CR is well-formed (clear AC, matching code surface, no governance conflicts)
- **Blocked CR:** Set status to `Blocked`, add `# Implementation Blockers` entries, hand to `@spec` agent
- **CR updates during implementation:** Document discoveries, update CR body, preserve intent, add revision history
- **Done:** Only when ALL acceptance criteria are individually verified

## 5. Test Rules

- **Self-sufficiency (ADR-051):** Tests must use only hardcoded inputs or fixtures under `tests/`. No dependency on `demo/`, `outputs/`, `tmp/`, or other mutable locations.
- **Controlled parameters:** Tests must set parameters explicitly for the path under test. Do not rely on mutable defaults unless testing defaults specifically.
- **Test-owned fixtures:** If a CR mentions demo files or corpus examples, those are NOT test inputs. Derive small test-owned fixtures.

## 6. Spec Writing Rules

- CRs should be the primary execution prompt — implementable from the CR itself + only its directly referenced records
- Use templates: `docs/internal/*/000-*-template.md`
- Run `python scripts/update-indexes.py` after adding/renaming governance records
- Use YAML front matter with lowercase snake_case keys

## 7. Coding Style

- Follow `docs/internal/coding-style.md` (moved from `.cline/coding-style.md`)
- Key: token-optimized, single quotes, short functions (≤20 lines), explicit comparisons, narrow exception handling

## 8. Code Index

- `docs/internal/code-index/module-tags.yaml` — domain tags for bounded code inspection
- `docs/internal/code-index/*_callgraph.mmd` — Mermaid call graphs (generated by `scripts/gen_code_graph.py`)
- `tests/dependency_map.yaml` — test marker routing

## 9. Agent Aliases

| Alias | File | Purpose |
|-------|------|---------|
| `@spec` | `.cline/spec.md` | Write/update CR, ADR, REQ, REVIEW docs |
| `@change` | `.cline/change.md` | Implement, verify, review CR-NNN |

## 10. Quick Commands

```bash
python -m pytest                          # full suite
python -m pytest tests/ -k "pattern"      # filtered
python scripts/update-indexes.py          # regenerate governance indexes
python scripts/gen_code_graph.py          # generate call graphs
```
