# 029. CLI Runtime Output via Logger Only

## Plain Summary

All CLI runtime output shall go through the shared logging system. Direct
standard `print()` output shall be reserved only for the built-in `--help`
response generated by argument parsing.

TL;DR: help may print; normal runtime must log.

## Context and Problem Statement

The package is moving toward centralized CLI logging through
`logging-actions`, shared option registration, and shared setup helpers in
`src/akkapros/lib/utils.py`. That design solves argument and logger setup, but
it still leaves an important policy question: whether ordinary runtime output
may continue using direct `print()` calls.

The user wants a strict rule because the tools may be used in batch mode. In
batch usage, raw `print()` statements create inconsistent console streams,
ignore shared output controls such as `--quiet` and `--no-console`, and make
logging behavior harder to reason about.

This ADR therefore defines a package-wide runtime-output rule across all CLIs.

## Decision Drivers

- Predictable batch-mode behavior
- Respect for shared output-control flags
- One consistent runtime-output path across all CLIs
- Easier debugging and review of console behavior
- Alignment with centralized logging adoption in ADR-028

## Considered Options

- Option A — Allow mixed `print()` and logger output after logging centralization. Rejected because batch behavior remains inconsistent and output-control flags become only partially effective.
- Option B — Reserve direct `print()` for `--help` only; all other runtime output goes through the logger. Chosen because it creates one enforceable rule and makes batch behavior predictable.
- Option C — Remove all console text including `--help` from standard printing. Not chosen because command-line help is naturally produced by `argparse` and does not belong to normal runtime logging.

## Decision Outcome

Choose Option B.

Across all CLI entrypoints, direct standard `print()` output shall be used only
for the `--help` response produced by the argument parser. All other runtime
messages, including:

- startup banners,
- progress messages,
- informational notices,
- warnings,
- errors surfaced by normal CLI flow, and
- batch-processing status

shall be emitted through the shared logger.

This rule applies to all CLIs in the package, including batch-capable tools and
monolithic command implementations such as `phoneprep`.

## Pros and Cons of the Options

### Chosen Option

- Pros: makes `--quiet` and `--no-console` semantically reliable
- Pros: improves batch-mode predictability
- Pros: simplifies reasoning about runtime output during debugging
- Pros: gives one clear migration rule for replacing existing `print()` calls
- Cons: requires touching many existing output call sites
- Cons: some current banner or progress output may need refactoring to fit the logger flow cleanly

### Other Options

- Option A:
  - Pro: smaller migration effort
  - Con: inconsistent runtime behavior remains
  - Con: direct prints bypass shared logging controls
- Option C:
  - Pro: maximal purity of output path
  - Con: unnecessary conflict with normal `argparse --help` behavior
  - Con: does not materially improve runtime logging over Option B

## Implications and Consequences

- Existing direct `print()` runtime output in CLI code should be migrated to logger calls.
- The shared logging helper layer must initialize early enough that startup messages can use it.
- Tests that assert console output may need to be updated to reflect logger-based behavior.
- `--help` remains a special case outside the runtime logging rule.
- Documentation for shared CLI behavior should explicitly distinguish parser help output from normal logged runtime output.
